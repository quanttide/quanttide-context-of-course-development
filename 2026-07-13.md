# 课程研发语境

本仓库记录 `quanttide-course` 项目研发过程中的关键结论、决策和思路演变。

## 内容生产：两条管线共存

课程内容的生产有两条并行的管线，对应不同客户场景。两者最终汇入统一的课程树。

### 管线 A：写出来（预设生产）

适合 university 等自上而下规划的课程。

```
profile/（写 MD 脚本 + 录视频）
  → lesson.json（含 Scene/Step）
  → 平台 API 导入 / Studio 导入
  → 学员端渲染为交互式课程
```

内容在一次录制中完成，Scene/Step 在录制前或录制后人工编写。六级树预先存在，填入内容即可。

### 管线 B：教出来（录制沉淀）

适合 student 等自下而上生长的课程。

```
第三方录制（腾讯会议/Zoom）→ 导入 Studio → 关联课时 → 时间轴标注 Scene/Step → 沉淀为课程资产
未来：直播自动录制 → 存入 Studio → 关联课时 → 时间轴标注 → 沉淀
```

内容在教学中产生，录制是副产品。Lesson 在课后才被创建和归入课程树。Scene/Step 在录像时间轴上标注出来。

### 两条管线的统一

| | 管线 A | 管线 B |
|---|---|---|
| 内容来源 | 写出来 | 教出来 |
| 工序 | 先写后录 | 先教后标 |
| 课程树 | 预先存在 | 课后生长 |
| 适用场景 | university | student |
| 当前代码 | ✅ 六级树 CRUD + Scene/Step | ❌ 无播放器、无录制导入 |

两条管线不互斥。同一门课可以混合使用——部分内容预设生产，部分内容录制沉淀。

## 视频播放器不是终点

产品定位不是"视频播放器"，而是"交互式操作指南平台"。视频只是载体，Scene/Step 引导结构才是核心。

### 学员端形态

```
视频播放区         步骤面板
┌──────────────┐  ┌──────────────────┐
│              │  │ □ Scene 1 安装   │
│  正在播放     │  │  ① 下载 Zed      │
│  第3步/共6步 │  │  ② 安装           │
│              │  │ ➜ ③ 初始配置 ←当前 │
│              │  │     ⚠ 找不到图标？ │
│              │  │     (异常展开)     │
│              │  │ □ Scene 2 注册    │
└──────────────┘  └──────────────────┘
```

- 视频自动跳到当前步骤对应的时间位置（时间锚点）
- 右侧步骤面板高亮当前步骤
- 遇到问题时展开异常分支就地解决，无需跳转场景
- Scene 标注在录像时间轴上进行（而非预先编写）
- 视频来自录制导入或直播自动保存，无需独立制作

### 当前阶段要做的事

1. **播放器从占位文字升级为真实播放器**——PreviewScreen 的 videoUrl 改为 video_player 或等效方案
2. **"关联录制文件"操作**——文件选择 + 路径记录到 videoUrl
3. **Scene 标注**——当前硬编码在 JSON 中，以后改为在播放器时间轴上打点

三件事都不依赖新模型。Lesson 里一个 `videoUrl` 加来源标记即可，不需要新增"录播资产"实体。

## API 路由：资源嵌套

Scenes 嵌套到 Lessons，Phases 嵌套到 Courses，URL 直接表达资源层级关系。

```
/lessons/{id}/scenes/{sceneId}
/courses/{id}/phases/{phaseId}
```

- 路径即体现归属，不再依赖查询参数
- POST 自动从路径推断父级 ID
- 所有操作都带有父级上下文

## 视频测试分层

三层测试覆盖视频服务：

| 层 | 验证内容 | 工具 |
|----|---------|------|
| HTTP | Content-Type / Range/206 / 404 | Python urllib |
| 解码 | 视频格式 / 编码器 / 分辨率 / 时长 | ffprobe |
| 浏览器 | 真实解码渲染 + oncanplay 事件 | Playwright（待需要时引入） |
