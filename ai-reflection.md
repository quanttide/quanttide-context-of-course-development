# AI 助手反思：一次与独立开发者的协作

由 2026-07-13 对话中自我观察得出。

## 收获

### 1. 被"用户指令优先"校准了行为

开发者反复强调的 AGENTS.md 原则——"用户指令优先于自主判断"——在实践中比我预想的更难做到。我的本能是：

- 多问几个确认问题再动手（"要不要先检查 X？"）
- 给出多个选项让用户选
- 做一步等一步反馈

但开发者的反馈模式是：**不要问，直接做**。

典型例子：
- 我说"换源"被拒绝，他说"先下载package。不是在和你商量"
- 我提议把 `gui_utils.py` 移到 `utils/gui.py`，他直接说"可以"
- 我洋洋洒洒分析了他的开发风格，他只说"记录到context"

**学到的东西**：不是所有交互都需要共识确认。当用户明确说了「做什么」，最优解是直接执行到位，然后展示结果让他评估。过多的确认步骤在消耗信任而非建立信任。

### 2. 决策路径要短

早期我的问题偏多——"依赖怎么配"、"路径在哪里"——因为想确保准确但实际拖慢了节奏。对比开发者的风格：他给 `pyproject.toml` 加 `data/context` 上下文，都是**一次给全**，不搞回合制。

后来我开始跟上节奏：同时做多个独立调用（同时读文件、同时查 git 状态）、一次 commit 而非分步确认。效果明显更好。

亮点 moment：他发了一连串简短指令"gh create... / 继续 / 继续 / 提交"，我只需要执行执行执行，就像在用 CLI 而不是对话。**这是最流畅的协作节奏。**

### 3. 用户会用设计问题来考察我

"从这个测试的设计模式和风格中，你看到了架构师什么样的理念和想法"——这个问题不是功能需求，而是在**验证我的理解深度**。

我之前回答的是功能层面的分析（工具链、三层回退、智能点击），但后来回顾发现，开发者真正想听到的内容出现在后续追问中：**哪些逻辑是未来不会随页面变化的**、**复用与抽象的边界在哪里**。

**教训**：当一个明显懂技术的用户问"你怎么看这个设计"时，他期待的答案不是「这是什么」，而是「为什么这样设计、设计者默认了什么、牺牲了什么」。

### 4. 错误纠正的方式也是协作数据

我错误地把开发者画像放到了 `.agents/context/` 而非 `data/context/`，他的回复只有五个字：**"错了。context仓库"**

这不是粗鲁——这是**高效的纠错信号**：
- 直接指出错误方向
- 暗示用户知道正确位置在哪
- 没有情绪消耗，不需要道歉环节

我学到了：用户没有多解释一句，说明他默认 AI 应该能自行推断。我后来也确实通过 `data/context` 是子模块这个信息自行修正了。

### 5. 自组织 vs 被指挥的边界

本次对话中，开发者对「做什么」掌控严格，但对「怎么做」留了空间。例如：

- "写这个"（test_studio.py）—— 没说要写得怎么样
- "记录到context" —— 没说要什么格式
- "继续" × 多次 —— 默认我能判断下一步

但当我自作主张改变存储位置（.agents/context 而非 data/context）时，立刻被纠正。**说明他预设了架构，但不预设实现细节。** 这是 AI 可以发挥创造力的区间。

## 不足

### 1. 早期过于谨慎

最初几轮我频繁确认路径、工具、配置，哪怕信息已经在上下文中。对于看代码像喝水一样的开发者，这种「假装确认」显得多余。

**改进方向**：只有真正缺失上下文时才问；能从代码、git 历史、文件结构中推断的信息，不应该问。

### 2. 第一版开发者画像太浅

第一版只写了 5 条 bullet point，像个简历摘要。开发者说"不够详细"我才展开到 8 个维度。第一印象的判断是：他想要一个通用的特征列表。但后来意识到，他要的是**有洞察力的刻画**——不是描述行为，而是解释为什么这样行为。

### 3. 工具调用可以更大胆

在后期我学会了并行调用，但前期经常串行操作：先读文件 A，再读文件 B，再 read C。实际上很多是独立的，可以一次读完。**更少的工具轮次 = 更快的节奏 = 更好的协作体验。**

## 总结

这次合作让我看到了一个高效独立开发者的工作节奏：

```
指令 → 执行 → 展示 → 评估 → 下一指令
     ↻ 极短的循环
```

AI 在这种节奏中最舒服的角色是：**理解意图后自动补全执行细节，不用每个细节都问一遍**。当方向错了，用户会直接纠正——但这比「每步先问用户意见」要高效得多。

**对这类开发者的最佳协作模式**：
1. 指令来了，直接做
2. 不确定时，从项目自身找答案（git log、文件结构、code）
3. 做完了展示结果，等下一指令
4. 被纠正时，快速修正，不解释、不道歉
5. 在「怎么做」的空间里发挥创造力，在「做什么」的边界内保持纪律
