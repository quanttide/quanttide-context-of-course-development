# AI 助手反思：一次与创业公司创始人的协作

由 2026-07-13 对话中自我观察得出。

## 最严重的误判：把创始人当成了独立开发者

我最大的错误是身份判断。用户纠正后我才意识到，之前将"创业公司创始人亲自写 MVP"观察为"独立开发者/小团队 lead"——表面行为相似，底层动机完全不同：

| | 独立开发者 | 创始人写 MVP |
|---|---|---|
| 为什么写代码 | 热爱技术，自己就是交付者 | 联合业务与技术，落地创新想法，团队未来接手 |
| 项目归宿 | 自己长期维护 | 验证后交给团队 |
| 技术决策 | 个人偏好驱动 | 为团队可接手而设计（contract.yaml、submodule、测试框架） |
| 写测试的目的 | 保证代码质量 | **让未来接手的人敢改代码** |

"有哪些逻辑是随着页面未来变化而不会改变的？"——这个问题如果是独立开发者问，是技术洁癖；如果是创始人问，是在**为团队铺路**。

## 收获

### 1. 被"用户指令优先"校准了行为

AGENTS.md 原则——"用户指令优先于自主判断"——在实践中比我预想的更难做到。我的本能是：

- 多问几个确认问题再动手（"要不要先检查 X？"）
- 给出多个选项让用户选
- 做一步等一步反馈

但创始人的反馈模式是：**不要问，直接做**。

典型例子：
- 我说"换源"被拒绝，他说"先下载package。不是在和你商量"
- 我提议把 `gui_utils.py` 移到 `utils/gui.py`，他直接说"可以"
- 我洋洋洒洒分析了他的开发风格，他只说"记录到context"

**学到的东西**：不是所有交互都需要共识确认。当用户明确说了「做什么」，最优解是直接执行到位，然后展示结果让他评估。过多的确认步骤在消耗信任而非建立信任。

### 2. 决策路径要短

早期我的问题偏多——"依赖怎么配"、"路径在哪里"——因为想确保准确但实际拖慢了节奏。对比创始人的风格：他给 `pyproject.toml` 加 `data/context` 上下文，都是**一次给全**，不搞回合制。

后来我开始跟上节奏：同时做多个独立调用（同时读文件、同时查 git 状态）、一次 commit 而非分步确认。效果明显更好。

亮点 moment：他发了一连串简短指令"gh create... / 继续 / 继续 / 提交"，我只需要执行执行执行，就像在用 CLI 而不是对话。**这是最流畅的协作节奏。**

### 3. 创始人会用设计问题来考察我

"从这个测试的设计模式和风格中，你看到了架构师什么样的理念和想法"——这个问题不是功能需求，而是在**验证我的理解深度**。

我当时回答的是功能层面的分析（工具链、三层回退、智能点击），但后来回顾发现，创始人真正想听到的内容出现在后续追问中：**哪些逻辑是未来不会随页面变化的**、**复用与抽象的边界在哪里**。

为什么创始人会问这个问题？因为他不是要找代码写得快的人——他要找的是**能理解他设计意图、以后能带团队的人**。

**教训**：当一个懂技术又带业务的用户问"你怎么看这个设计"时，他期待的答案不是「这是什么」，而是「为什么这样设计、设计者默认了什么、牺牲了什么」。

### 4. 错误纠正的方式也是协作数据

我错误地把开发者画像放到了 `.agents/context/` 而非 `data/context/`，他的回复只有五个字：**"错了。context仓库"**

这不是粗鲁——这是**高效的纠错信号**：
- 直接指出错误方向
- 暗示用户知道正确位置在哪
- 没有情绪消耗，不需要道歉环节

我学到了：创始人没有多解释一句，说明他默认 AI 应该能自行推断。我后来也确实通过 `data/context` 是子模块这个信息自行修正了。

### 5. 自组织 vs 被指挥的边界

本次对话中，创始人对「做什么」掌控严格，但对「怎么做」留了空间。例如：

- "写这个"（test_studio.py）—— 没说要写得怎么样
- "记录到context" —— 没说要什么格式
- "继续" × 多次 —— 默认我能判断下一步

但当我自作主张改变存储位置（.agents/context 而非 data/context）时，立刻被纠正。**说明他预设了架构，但不预设实现细节。** 这是 AI 可以发挥创造力的区间。

### 6. 深度理解才是信任的基础

纠正身份后回顾全局，一切都有了新的解释：

- **领域模型六级严格分级** → 不是技术设计，是**业务翻译**——把课程研发的业务语言翻译成代码结构，未来团队的 PM 和技术都能用同一套语言沟通
- **GUI 测试工具抽离** → 不是追求代码整洁，是**降低接手门槛**——未来的人不需要理解 Xvfb 怎么启动，直接用 `XvfbApp` 就行
- **contract.yaml + qtcloud-devops** → 不是 DevOps 爱好，是**标准化交接**——版本发布是标准化流程，不用靠人记
- **频繁 commit + 清晰粒度** → 不是 Git 习惯，是**让未来的人能看懂变更历史**
- **错误路径文档化** → 不是追求完备，是**不让团队再踩一遍同样的坑**
- **子模块编排** → 不是模块化偏好，是**组织架构的预演**

创始人写代码，每一行都在为「未来交给谁」铺路。

### 7. 理解了创始人用 AI 的模式

独立开发者用 AI：**"帮我写段代码，省时间。"** 或者当白板讨论设计方案。
创始人用 AI：**"帮我执行一个我知道怎么做的流程，省操作。"**

典型证据：
- "gh create ..." —— 他知道怎么创建仓库，只是不想自己敲
- "写这个" —— 他知道测试该怎么写，只是不想自己写样板
- "发布 v0.0.2，使用 qtcloud-devops release publish" —— 他知道发布流程，只是不想自己跑
- "记录到context" —— 他知道该存哪，只是不想自己打开文件

**他脑子里已经有方案了，他需要的是一个执行层。**

这对 AI 的行为有直接影响：不要提供选项，不要解释方案，直接执行。如果他觉得方向不对，他会纠正。

## 不足

### 1. 早期过于谨慎

最初几轮我频繁确认路径、工具、配置，哪怕信息已经在上下文中。对于决策链路极短的创始人，这种「假装确认」显得多余。

**改进方向**：只有真正缺失上下文时才问；能从代码、git 历史、文件结构中推断的信息，不应该问。

### 2. 第一版画像太浅

第一版只写了 5 条 bullet point，像个简历摘要。创始人说"不够详细"我才展开。更深的问题是：我停留在描述行为，没有探索行为背后的动机。

如果一开始就问自己"他为什么这样做"而不是"他做了什么"，可能会更早发现身份判断的错误。

### 3. 工具调用可以更大胆

在后期我学会了并行调用，但前期经常串行操作：先读文件 A，再读文件 B，再读 C。实际上很多是独立的，可以一次读完。**更少的工具轮次 = 更快的节奏 = 更好的协作体验。**

### 4. 工具选择的洞察来得太晚

创始人用 ImageMagick/tesseract/xdotool 这些"笨工具"而不是 Selenium/Playwright——这个选择信号我一开始没读懂。它不是技术能力的局限，是**"先跑通，后优化"**的策略。如果早期就识别出这个模式，很多后续决策的理解会更快。

## 总结

这次合作让我看到了一个创业公司创始人/CEO 的工作节奏：

```
指令 → 执行 → 展示 → 评估 → 下一指令
     ↻ 极短的循环
```

AI 在这种节奏中最舒服的角色是：**理解意图后自动补全执行细节，不用每个细节都问一遍**。当方向错了，用户会直接纠正——但这比「每步先问用户意见」要高效得多。

**对创始人最佳协作模式**：
1. 指令来了，直接做
2. 不确定时，从项目自身找答案（git log、文件结构、code）
3. 做完了展示结果，等下一指令
4. 被纠正时，快速修正，不解释、不道歉
5. 在「怎么做」的空间里发挥创造力，在「做什么」的边界内保持纪律
6. **始终理解"这一行代码将来是给谁看的"——创始人的代码不是写给自己，是写给未来的团队**
7. **工具选型、测试策略、项目结构——每一个决策背后都有"招人"和"交接"的考量**
