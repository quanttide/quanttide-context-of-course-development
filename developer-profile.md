# 创始人画像

创业公司创始人/CEO，正在亲自搭建 qcloud-course MVP，验证后交给团队接手。

## 核心特征

**行为的根本动机**：创始人的每一行代码都是写给未来团队看的，不是为了自己。

这一动机体现在所有决策中：

| 表面行为 | 深层动机 |
|---------|---------|
| 领域模型六级严格分级 | 把业务语言翻译成代码结构，未来 PM 和开发用同一套语言沟通 |
| GUI 测试工具抽离为 `XvfbApp` | 降低接手门槛，新人不需要懂 Xvfb |
| `contract.yaml` + qtcloud-devops 发布流水线 | 标准化交接流程，不用靠人记 |
| 频繁 commit + 清晰粒度 | 让未来的人能看懂变更历史 |
| 错误路径也文档化 | 不让团队重新踩一遍坑 |
| 子模块编排 | 预演组织架构和团队分工 |
| 多条技术方案并存而非只留最优 | 保留选择空间，接手的人可能环境不同 |

## 工作特征

### 产品 + 基础设施并行推进

同时建设产品功能（六级树、Scene/Step、Studio）和基础设施（GUI 测试框架、devops 流水线、submodule 编排），两条线在同一次开发中交替推进。

### 决策模式：快与准的分界清晰

- **该快时**："先下载 package。不是在和你商量"、"有没有更快的办法"
- **该准时**：写测试前先分析"哪些逻辑会变哪些不会"、发布流程严格按 contract 走

### 技术栈选型含"招人"考量

| 选择 | 理由 |
|------|------|
| Flutter（非 Tauri） | 生态大，招 Dart 开发者相对容易 |
| Python/pytest（非任意语言） | 国内 Python 开发者最多 |
| ImageMagick/tesseract/OpenCV（笨工具） | 现成的，未来团队不用维护新依赖 |

"够用"和"好招人"之间取平衡，不走最极致方案。

### 沟通与协作习惯

- 指令直接："gh create"、"写这个"、"提交"——不需要铺垫
- 纠错高效："错了。context 仓库"——两句话指清错误 + 正确位置
- 给上下文但不替 AI 想：引用多个文件让 AI 自己判断
- 用设计问题考察理解：问"你看到了什么理念"而非"这个功能怎么做"

### 对 AI 的使用模式

创始人用 AI**不是**当白板讨论方案，而是当执行层：

- "gh create"——知道怎么创建仓库，不想自己敲
- "写这个"——知道测试怎么写，不想写样板
- "发布 v0.0.2"——知道发布流程，不想自己跑
- "记录到 context"——知道该存哪，不想打开文件

**脑子里已经有方案了，需要的是执行层。**

## 关键差异速查：创始人 vs 独立开发者

| 维度 | 独立开发者 | 创业公司创始人 |
|------|-----------|---------------|
| 为什么写代码 | 热爱技术，自己就是交付者 | 联合业务与技术，团队未来接手 |
| 项目归宿 | 自己长期维护 | 验证后交给团队 |
| 技术栈选择 | 个人偏好/技术前沿 | 好招人 + 够用 |
| 测试动机 | 保证代码正确 | 降低接手门槛 |
| 项目结构 | 个人习惯 | 预演组织架构 |
| 工具选择 | 统一方案/最新方案 | 最快能跑的方案 |
| 文档风格 | 怎么用 | 为什么这么做 + 为什么不那么做 |
| 失败记录 | 自己记住 | 写进文档，不让团队再踩 |
| AI 使用 | 白板/讨论伙伴 | 键盘/执行层 |
| 知识封装 | 可复用的代码 | 可传递的知识 |
