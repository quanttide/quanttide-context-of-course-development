# 开发者画像

由 2026-07-13 对话中观察得出。

## 身份

**创业公司创始人/CEO**，在 building **量潮课程云（qtcloud-course）**。正在亲自搭建早期 MVP，等产品验证后交给团队接手。亲自写代码不是因为缺人，而是需要**联合业务与技术**，并且有大量创新想法要在架构层面落地。

## 工作特征

### 1. 产品 + 基础设施并行推进

- **产品线**：课程研发教程仓库（`quanttide-tutorial-of-course-development`）、课程蓝图、视频录制计划（Lesson→Sense→Step 分级体系）、Flutter 桌面客户端（qtcloud_course_studio）
- **基础设施线**：GUI 自动化测试框架（`utils/gui.py`，支持 Xvfb/OCR/OpenCV）、`qtcloud-devops` 发布流水线、submodule 编排、contract.yaml 组件化管理
- 两条线在同一个 session 中交替推进，互不偏废

### 2. 领域模型驱动的设计思维

课程内容有六级严格分级，每级对应不同的视频时长和交付粒度：

```
Program (专业)     → 大数据微专业
  Course (课程)    → 氛围编程
    Phase (阶段)    → 安装编程工具
      Lesson (课时) → 安装编程环境 (30-40min)
        Sense (场景) → 配置Zed (5-10min)
          Step (步骤) → 打开设置页面 (1-2min)
```

这不是临时想出来的，是提前设计好的领域模型。每个层级可以独立替换，支持正常路径和异常路径。

这个领域模型更大的价值在于**业务翻译**——把课程研发的业务语言翻译成代码结构，未来团队的 PM 和技术都能用同一套语言沟通。

### 3. 分层抽象的工程直觉

代码设计同样体现分层思维：

- **全局分析** "有哪些逻辑是随着页面未来变化而不会改变的？" —— 提前识别稳定层与易变层
- **工具抽离**：`gui_utils.py` → `utils/gui.py`，把通用能力（窗口查找、Xvfb 生命周期、截图）封装为独立模块
- **三种定位策略分层**：OCR bounding box → 像素颜色探测 → OpenCV 模板匹配，按场景选最优
- `smart_click` 模式：首屏 OCR 自定位 → 缓存模板 → 后续模板匹配，替代硬编码坐标

动机不只是代码整洁——**是降低接手门槛。** 未来的人不需要理解 Xvfb 怎么启动，直接用 `XvfbApp` 就行。

### 4. 创始人写 MVP 的动机

创始人亲自写代码，和独立开发者写代码，表面行为相同但底层动机不同：

- **联合业务与技术**：课程云涉及课程研发、教学管理、视频分发等业务逻辑，这些不是纯技术需求——需要有人把业务领域模型翻译成代码架构。创始人是这个翻译官。
- **创新想法快速落地**：有大量创新想法需要在架构层面落地，交给别人做需要先解释清楚再等反馈，自己动手是最短路径。

### 5. 决策模式：快与准的分界清晰

对话中多次体现「该快则快，该准则准」的分界感：

**该快时：**
- "先下载package。不是在和你商量" —— 不讨论必要性，直接执行
- "有没有更快的办法"、"换源" —— 主动优化执行路径
- 多步操作并行驱动（同时启动 Xvfb、截图、OCR）

**该准时：**
- 写测试前先读 studio.md 理解应用结构
- GUI 测试工具设计前先分析哪些逻辑会变、哪些不会
- 发布流程严格按 contract.yaml → CHANGELOG → tag → Release 走

### 6. 技术栈选择背后的招人考量

创始人选技术栈不只是看技术好不好，还要看市场上好不好招人。对比独立开发者的选择差异：

| 技术栈 | 独立开发者可能选 | 创始人选了 |
|--------|----------------|-----------|
| 桌面端 | Tauri/Rust（性能，酷） | Flutter（生态大，招 Dart 开发者相对容易） |
| CLI | 任意 | Rust（但没用更小众的 Zig/Nim） |
| 测试 | 够用就行 | Python/pytest（国内 Python 开发者最多） |
| 系统工具 | 自己写 | ImageMagick/tesseract/OpenCV（现成的，未来团队不用维护） |

Flutter 桌面端在国内还算小众，但相比 Tauri 生态更成熟。这个选择说明：**在"够用"和"好招人"之间取了平衡，没走最极致的方案。**

### 7. 测试策略的多层动机

创始人写测试，从对话中的细节看有多个层次：

- 独立开发者写测试：**保证我现在写的代码不出错。**
- 创始人写测试更多是为了：
  - "有哪些逻辑是随着页面未来变化而不会改变的？" → 识别稳定层，让未来的人只改易变层
  - "有没有可以复用的测试工具" → 减少未来团队的重复工作
  - smart_click 模式 + 模板缓存 → 新员工不懂 Xvfb 也能写测试
  - 把错误路径也文档化 → "这个方法我们试过不行，别走这条路"

独立开发者不会把"哪些方法试过不行"写进文档——他自己踩过就知道了。创始人写进去，是因为**不想让未来团队里的每个人重新踩一遍同样的坑。**

### 8. 子模块策略：组织架构的预演

项目大量使用 git submodule 来拆分关注点：

- `apps/qtcloud-course` —— 主代码子模块
- `data/context` —— 对话上下文
- `data/profile` —— 课程资料
- `examples/default` —— 示例构建产物

子模块模式的好处不只是代码组织：

- 团队可以一个人负责 `provider`（Go），另一个人负责 `studio`（Flutter），互不干扰
- 各自有独立的版本号、CI、CHANGELOG
- 父仓库像"地图"，新人看一眼就知道项目有哪些组件

这个结构不是技术偏好，是**组织架构的预演**。在代码结构里画好了未来的团队分工。不追求 monorepo 大一统。

### 9. "低端工具"策略：先跑通，后优化

创始人用了 ImageMagick 的 `import` 和 `convert`、tesseract 的 OCR、系统 xdotool——这些都是"笨工具"。

独立开发者可能倾向于 Python + Selenium/Playwright 统一方案，或者自己用 OpenCV + PyTesseract 封装一套。

创始人选择的是**最直接能跑的方案**：系统有什么用什么，不引入新的运行时依赖。等验证了方案可行，再决定要不要花时间封装。

这和"先下载package。不是在和你商量"的逻辑一致：**先跑通，后优化。** 独立开发者可能更倾向于先选好方案再动手。

### 10. 知识的封装粒度

创始人封装知识的方式和独立开发者不同：

- 他把测试踩坑写成一篇文章（gui-automation-testing.md），而不是只留一个 `gui_utils.py`
- 他把课程分级做成一个领域模型，不只是代码里的类型定义
- 他把发布流程定成 contract.yaml，不只是自己写个脚本
- GUI 测试设计时保留了三种定位策略并写了对比表，没有只留推荐方案

**他在封装"可传递的知识"，不只是"可复用的代码"。** 代码可以复用，但团队需要的是理解。文档、模型、契约——这些是比代码更粗粒度的知识载体。

为什么保留所有方案而只删最优？因为**未来接手的人可能不具备安装 OpenCV 的环境，或者 OCR 更快更适合他们的场景。** 留下选择空间，比替团队做决定更负责。

### 11. 沟通与协作习惯

- **指令清晰直接**："gh create ..."、"写这个"、"提交" —— 不需要铺垫
- **纠错效率高**："错了。context仓库" —— 两句话告诉 AI 错了 + 正确位置
- **给上下文但不替 AI 想**：引用多个文件作为上下文，让 AI 自己判断
- **用设计问题考察理解**："从这个测试的设计模式和风格中，你看到了架构师什么样的理念和想法" —— 在验证 AI 的理解深度
- **commit 频率高**，每次提交粒度适中（一个逻辑变更一个 commit）

### 12. 对 AI 的使用模式

创始人使用 AI 的方式和独立开发者不同：

- 独立开发者用 AI：**"帮我写段代码，省时间。"** 或者当白板讨论设计方案。
- 创始人用 AI：**"帮我执行一个我知道怎么做的流程，省操作。"**

典型证据：
- "gh create ..." —— 他知道怎么创建仓库，只是不想自己敲
- "写这个" —— 他知道测试该怎么写，只是不想自己写样板
- "发布 v0.0.2，使用 qtcloud-devops release publish" —— 他知道发布流程，只是不想自己跑
- "记录到context" —— 他知道该存哪，只是不想自己打开文件

**他脑子里已经有方案了，他需要的是一个执行层。** 独立开发者更常把 AI 当白板（"帮我想想这个怎么设计"），创始人把 AI 当键盘（"我告诉你做什么，你做"）。

## 沟通偏好

- 用户指令优先于自主判断，先执行用户要求的方案，后提供改进建议
- 方案选择权在用户，不在 AI
- 纠正错误时直接指出，不需要额外铺垫

## 创始人 vs 独立开发者：关键差异速查

| 维度 | 独立开发者 | 创业公司创始人 |
|------|-----------|---------------|
| 为什么写代码 | 热爱技术，自己就是交付者 | 联合业务与技术，团队未来接手 |
| 项目归宿 | 自己长期维护 | 验证后交给团队 |
| 技术栈选择 | 个人偏好/技术前沿 | 好招人 + 够用 |
| 测试动机 | 保证代码正确 | 降低接手门槛 |
| 项目结构 | 个人习惯 | 预演组织架构 |
| 工具选择 | 统一方案/最新方案 | 最快能跑的方案 |
| 文档风格 | 怎么用 | 为什么这么做 + 为什么不那么做 |
| 失败记录 | 自己记住 | 写进文档，不让团队再踩 |
| AI 使用 | 白板/讨论伙伴 | 键盘/执行层 |
| 知识封装 | 可复用的代码 | 可传递的知识 |
