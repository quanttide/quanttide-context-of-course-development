# 从客户案例反观产品设计

两个案例（student / university）几乎是教学场景的两极，暴露了当前产品设计中未被检验的假设。

## 案例对比

| 维度 | Student | University |
|------|---------|------------|
| 课程来源 | 按需生长，课后补结构 | 预先规划，先搭骨架 |
| 教学方式 | 实操为主，讲师引导 | 理论授课 + 实践 |
| 考核 | 功能跑通即通过 | 评分标准 + 五档样本 |
| 班级规模 | 1 人 | 40 人 |
| 教师 | 1 人 | 多人（企业 + 校内） |
| 节奏 | 灵活，按学员进度 | 固定学期制 |

## 产品当前的隐性偏向

当前产品设计在**无意识地偏向 university 模式**：

- 六级树天然适合自上而下规划
- ContentStatus 控制发布流程（适合有版本管理的正式课程）
- Class 模型预设了多人的教务管理场景
- Assessment 模型（maxScore + passingScore）更接近连续评分

Student 场景需要的"先有叶子再长树干"的自下而上路径，产品有规划（独立课时池 + 归入阶段）但尚未实现。

## 考核模块需要重新审视

- Student：作业即验收，没有考试，评分是二元的（通过/不通过）
- University：有作业有考试，评分是连续的（50-90 分档），有详细评分标准

当前 Assessment 模型拿它去套 student 的考核场景会过度工程化——一个二元验收不需要满分和及格线。可能需要考虑 AssessmentType 不同时裁剪字段，或允许评分模式切换（pass/fail vs numeric）。

## 案例集的价值

案例不是为了确认产品做对了，而是暴露未检验的假设：

- 单人班级的 UI 应该长什么样？
- 自下而上的课程编辑路径在当前的树状 UI 中是否可行？
- 二元的考核通过/不通过是否需要单独的 AssessmentType？
- 多个教师角色（企业讲师 vs 校内教师）如何区分？

## 结论

三个模块集成确实是业务需要的，但每个模块在不同场景下的权重和表现方式完全不同。产品设计需要允许同一个实体在不同客户场景下呈现不同的复杂度。
